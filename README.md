# Бот Rubik — документация 

Единый бот для **Telegram** и **MAX**: регистрация (в т.ч. через AD), заявки в ТП (сайт/Лупа, WMS, смена пароля), уведомления. Админы: удаление пользователей, отчёт Лупа (Excel). Изменение учётных данных через бота отключено по требованию ИБ.

---

## 1. Реализация и архитектура кода

### 1.1 Разделение слоёв

- **core/** — бизнес-логика без привязки к каналу: регистрация, создание заявок, Jira, уведомления. Не импортирует aiogram или MAX SDK.
- **handlers/** — обработчики Telegram: вызов core API, рендер ответа в сообщения/кнопки.
- **adapters/max/** — то же для MAX: маршрутизация callback и сообщений, вызов того же core API.
- **user_storage.py** — единственное место хранения профилей и индексов (файлы в `data/`). Все обращения к персональным данным идут через него.

Зависимости направлены внутрь: handlers и adapters зависят от core и config; core не зависит от каналов.

### 1.2 Поток данных при действии пользователя

1. Входящее событие (callback или сообщение) приходит в `main.py` (TG) или `adapters/max/main_max.py` (MAX).
2. Определяется `user_id` канала и действие (callback_data или текст).
3. Вызывается соответствующий обработчик; он обращается к **core.support.api** (например, `get_main_menu`, `create_ticket`, `get_my_tickets`).
4. Core проверяет регистрацию (`user_storage.is_user_registered`), при необходимости — права (`config.is_channel_admin`, `is_lupa_report_allowed`), читает/пишет данные через **user_storage** и внешние API (Jira, AD/LDAP).
5. Ответ (Menu / Text / Error или результат создания заявки) возвращается адаптеру; адаптер формирует сообщение и кнопки и отправляет в канал.

### 1.3 Ключевые файлы реализации

| Назначение | Файлы |
|------------|--------|
| Точка входа, запуск TG и MAX | `main.py` |
| Конфигурация (только из env) | `config.py` |
| Хранение профилей и индексы | `user_storage.py` |
| Валидация ввода | `validators.py` |
| Единый API для каналов | `core/support/api.py`, `core/support/models.py` |
| Регистрация (в т.ч. из AD) | `core/registration.py`, `core/ad_ldap.py`, `handlers/registration.py` |
| Создание заявок (Jira) | `core/jira_aa.py`, `core/jira_wms.py`, `core/jira_lupa.py`, `core/support/api.py` (create_ticket) |
| Смена пароля | `core/password.py`, `handlers/password.py` |
| Реестр заявок и уведомления | `core/support/issue_binding_registry.py`, `core/notifications.py` |
| Админ: удаление пользователя, отчёт | `handlers/admin.py`, `core/lupa_report.py` |
| MAX: маршрутизация и сценарии | `adapters/max/main_max.py`, `adapters/max/handlers.py`, `wms_flow.py`, `lupa_flow.py` |

## 2. Проверки и защита

### 2.1 Отсутствие секретов в коде

- Все токены, пароли, URL и чувствительные параметры задаются **только через переменные окружения** (файл `.env`, не коммитируется: см. `.gitignore`).
- В коде нет хардкода логинов/паролей Jira, Telegram, MAX, AD. В примерах конфигурации используются нейтральные плейсхолдеры (например, `user@stdp.ru`).
- Файл `.env` и варианты `.env.*` перечислены в `.gitignore`; в репозиторий попадает только шаблон (например, `.env.example` без реальных значений).

### 2.2 Контроль доступа

- **Регистрация и базовые действия**: доступны только после прохождения регистрации или привязки аккаунта. Проверка: `user_storage.is_user_registered(user_id, channel_id)`.
- **Админ-панель и удаление пользователей**: только для полных админов. Список ID задаётся в env: `ADMIN_IDS` (Telegram), `ADMIN_MAX_IDS` (MAX). Проверка: `config.is_channel_admin(channel_id, user_id)`.
- **Отчёт Лупа (Excel)**: доступен полным админам и отдельно заданным «админам Лупы»: `ADMIN_LUPA_IDS`, `ADMIN_LUPA_MAX_IDS`. Проверка: `config.is_lupa_report_allowed(channel_id, user_id)`.
- Перед созданием заявки, просмотром «Мои заявки», добавлением комментария проверяется как регистрация, так и (где нужно) принадлежность заявки пользователю (`support_api.user_owns_issue`).

### 2.3 Валидация ввода

- **validators.py**: централизованная проверка ФИО (кириллица), рабочего логина, корпоративной почты (@petrovich.ru / @petrovich.tech), телефона, табельного номера. Используются при регистрации, при создании заявок и (где оставлено) при изменении данных.
- Регистрация через AD: введённая почта проверяется на формат; при привязке по контакту проверяется формат телефона. По номеру телефона из контакта выполняется поиск в AD (атрибуты telephoneNumber, mobile, ipPhone) — запись должна существовать.
Дополнительно в Telegram и MAX сравниваются contact.user_id и ID пользователя, который проходит регистрацию; бот принимает только собственный контакт пользователя. Перед сохранением профиля дополнительно проверяется совпадение введённой почты с полем mail в найденной записи AD (и номер телефона, и почта таким образом проверяются).
- Дубликаты: перед сохранением профиля проверяются уникальность логина, почты, табельного номера и телефона (`user_storage`: `check_login_or_email_taken`, `check_employee_id_taken`, индексы по логину/email/телефону).

### 2.4 Хранение и защита персональных данных

- Профили пользователей хранятся в `data/user_data.json`. Каталог `data/` и файлы с персональными и служебными данными перечислены в `.gitignore` и не попадают в репозиторий.
- Опционально включено шифрование полей профиля (ФИО, логин, почта, телефон) в покое: переменные `ENCRYPT_USER_DATA=1` и `USER_DATA_ENCRYPTION_KEY` (см. `user_storage.py`). Ключ хранится только в окружении.
- Индексы (по логину, email, телефону, табельному номеру, привязке MAX) строятся из расшифрованных данных и хранятся в `data/`; при шифровании в файл пишутся уже зашифрованные значения профилей.

### 2.5 Регистрация и обработка почты/телефона

- Регистрация в Telegram и MAX: запрашивается рабочая почта и контакт (телефон). По телефону выполняется поиск в AD (LDAP, только чтение); при совпадении почты с AD профиль создаётся из данных AD. Подразделение из AD не сохраняется в профиль; подразделение для заявок запрашивается при первом создании заявки из списка Jira.
- **Удаление сообщения с почтой**: по требованию ИБ после ввода почты при регистрации сообщение пользователя с почтой удаляется:
  - в Telegram: в `handlers/registration.py` после валидации и сохранения почты в state вызывается `message.delete()`;
  - в MAX: в `adapters/max/main_max.py` после успешной обработки email при регистрации из апдейта извлекается id сообщения пользователя и вызывается удаление сообщения через API MAX (если доступно).


### 2.6 Логирование и аудит

- Логирование настраивается в `main.py` (уровень через `LOG_LEVEL`); логи пишутся в консоль и при необходимости в `data/bot.log`. Файлы логов в `data/` в `.gitignore`.
- В коде при критичных действиях (регистрация, удаление пользователя админом, создание заявок) выполняется логирование (например, `logger.info` с идентификаторами пользователя и типа операции). Логи не содержат паролей и токенов; при необходимости в логах могут фигурировать только идентификаторы (user_id, issue_key).

### 2.7 Внешние интеграции

- **Jira**: доступ по токену/учётным данным из env; запросы к Jira AA, PW (WMS), WHD (Лупа) — создание задач, статусы, комментарии. Данные пользователя (ФИО, логин, телефон, подразделение) передаются только в объёме, необходимом для создания заявки.
- **AD/LDAP**: только чтение; используется для проверки пользователя по номеру телефона при регистрации. Учётные данные подключения к AD задаются в env; в коде нет дефолтных URL и паролей.

## 4. Установка и запуск

- Python 3.10+.
- Зависимости: `pip install -r requirements.txt`.
- Конфигурация: копирование `.env.example` в `.env` и заполнение переменных (токены, ID админов, Jira, при необходимости AD). Файл `.env` не коммитируется.
- Запуск: `python main.py`. Остановка: `Ctrl+C`.

---

## 5. Роли и права

- **Обычный пользователь**: регистрация/привязка, создание заявок, мои заявки, помощь. Без доступа к админ-панели и чужим данным.
- **Полный админ** (`ADMIN_IDS` / `ADMIN_MAX_IDS`): админ-панель, удаление пользователей, отчёт Лупа.
- **Админ Лупы** (`ADMIN_LUPA_IDS` / `ADMIN_LUPA_MAX_IDS`): только кнопка «Отчёт Лупа (Excel)», без удаления пользователей.
