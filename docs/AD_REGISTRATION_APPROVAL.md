# Проверка пользователя в AD при регистрации в боте

Документ для согласования с серверным админом: как бот проверяет сотрудника в Active Directory и какие данные запрашивает/выгружает.

---

## 1. Назначение

При регистрации в Telegram-боте пользователь не вводит ФИО, логин и подразделение вручную. Бот **один раз** обращается к AD по LDAP/LDAPS, находит запись сотрудника по номеру телефона и **только читает** нужные атрибуты. По ним формируется профиль в боте. Запись в AD не изменяется.

---

## 2. Что запрашивается у пользователя (в боте)

| Шаг | Что просит бот | Зачем |
|-----|----------------|-------|
| 1   | **Рабочая почта** (текст, @petrovich.ru / @petrovich.tech) | Проверка при поиске: почта из AD должна совпадать с введённой. |
| 2   | **Контакт (телефон)** — кнопка «Поделиться контактом» в Telegram | Получение реального номера телефона для поиска в AD. |

Другие данные (ФИО, логин, подразделение) у пользователя **не запрашиваются** — они берутся только из AD.

---

## 3. Подключение к AD

- **Протокол:** LDAP по TLS (LDAPS), порт **636** (рекомендуется) или обычный LDAP **389**.
- **Учётная запись для подключения:** одна **служебная учётная запись** (service account), логин и пароль которой задаются в конфигурации бота (переменные окружения). Пароль не хранится в коде, только в конфиге на сервере.
- **Права учётной записи:** достаточно прав **только на чтение** атрибутов пользователей в указанном поддереве (BASE_DN). Запись в AD бот не выполняет.
- **Проверка сертификата:** при использовании самоподписанного сертификата на контроллере допускается отключение проверки (настраивается отдельной переменной).

---

## 4. Поиск пользователя в AD

- **Основа поиска (BASE_DN):** задаётся в конфигурации (например, `OU=Office_Users,OU=PTech,OU=stdp,DC=stdp,DC=ru`). Поиск выполняется в этом поддереве (search scope: subtree).
- **Критерий поиска:** номер телефона. Номер, полученный из Telegram, нормализуется до цифр (последние 10 цифр номера). Поиск выполняется по атрибутам:
  - **telephoneNumber**
  - **mobile**
  - **ipPhone**
- **Фильтр LDAP:** логическое «ИЛИ» по совпадению подстроки с этими цифрами в любом из трёх атрибутов, например:  
  `(|(telephoneNumber=*XXXXXXXXXX*)(mobile=*XXXXXXXXXX*)(ipPhone=*XXXXXXXXXX*))`  
  где `XXXXXXXXXX` — 10 цифр номера (без ведущей 7).
- Если найдено несколько записей — используется **первая**. Если ни одной — пользователю показывается сообщение о том, что в базе сотрудников он не найден, и ссылка на портал поддержки (если задана в конфиге).

---

## 5. Запрашиваемые атрибуты AD (только чтение)

Бот запрашивает у найденной записи пользователя **только** следующие атрибуты (read-only):

| Атрибут AD            | Назначение в боте |
|----------------------|-------------------|
| **displayName**       | ФИО (приоритет). Запасной вариант — **cn**. |
| **cn**                | ФИО, если displayName пуст. |
| **sAMAccountName**    | Рабочий логин (логин входа в домен). |
| **mail**              | Корпоративная почта (приоритет). |
| **userPrincipalName** | Почта, если атрибут mail пуст. |
| **telephoneNumber**   | Номер телефона (один из источников для поиска и для профиля). |
| **mobile**            | Мобильный телефон (поиск и профиль). |
| **ipPhone**           | Телефон (поиск и профиль). |
| **department**        | Подразделение (для отображения в боте). |
| **title**             | Должность (запрашивается, при необходимости можно использовать в боте). |

Остальные атрибуты объекта пользователя **не запрашиваются и не обрабатываются**.

---

## 6. Что выгружается и сохраняется в боте

После успешного поиска в AD бот формирует **внутренний профиль пользователя** в своей базе (файлы в каталоге `data/` на сервере бота). В этот профиль попадают **только** приведённые к единому виду значения:

| Поле профиля в боте | Источник в AD |
|---------------------|---------------|
| **full_name**       | displayName или cn |
| **login**           | sAMAccountName (в нижнем регистре) |
| **email**           | mail или userPrincipalName (в нижнем регистре) |
| **phone**           | Нормализованный номер из mobile / telephoneNumber / ipPhone или из контакта Telegram |
| **department**      | department (если заполнен в AD) |

Эти данные используются только для работы бота (меню, заявки, смена пароля и т.п.). В AD **ничего не записывается и не изменяется**.

---

## 7. Проверка совпадения почты

После поиска по телефону бот проверяет: **почта из записи AD** (атрибут mail или userPrincipalName) должна **совпадать** (без учёта регистра) с **почтой, которую пользователь ввёл на шаге 1**. Если не совпадает — регистрация не завершается, пользователю предлагается ввести почту заново или поделиться контактом с другого номера.

---

## 8. Итог для апрува

- Один **только читающий** доступ к AD по LDAPS (или LDAP) с одной служебной учётной записью.
- Поиск выполняется **только по номеру телефона** (атрибуты telephoneNumber, mobile, ipPhone) в заданном BASE_DN.
- Запрашиваются **только** перечисленные выше атрибуты (displayName, cn, sAMAccountName, mail, userPrincipalName, telephoneNumber, mobile, ipPhone, department, title).
- Из AD **ничего не изменяется**; выгружаются только данные для формирования профиля в боте (ФИО, логин, почта, телефон, подразделение).

Если такой режим доступа устраивает, можно выдавать боту учётную запись с правами **только на чтение** в указанном поддереве и разрешить исходящие подключения с сервера бота к контроллеру по порту 636 (LDAPS) или 389 (LDAP).
